# Maven / Gradle 本地依赖缓存旧版本清理工具需求分析

- **文档名称**：Maven / Gradle 本地依赖缓存旧版本清理工具需求分析
- **适用项目**：本地开发工具 / 磁盘清理工具 / Java 工程效率工具
- **创建日期**：2026-03-27
- **适用平台**：macOS、Windows

---

## 1. 背景

在 Java / Spring / Maven / Gradle 日常开发中，本地依赖缓存会随着项目迭代不断累积依赖包。  
同一个依赖往往会存在多个历史版本，例如：

- `6.1.2`
- `6.1.5`
- `6.1.6`
- `6.1.13`
- `6.1.14`
- `6.1.16`
- `6.2.1`
- `6.2.3`
- `6.2.10`
- `6.2.11`
- `6.2.12`
- `7.0.2`

随着时间推移，这些历史版本会在本机磁盘中占据大量空间，尤其是长期维护多个项目、多个分支，或频繁切换技术栈的开发者，本地缓存体积会持续膨胀。

因此，存在一个很明确的需求：

> 开发一款产品，用于扫描本机 Maven repository 与 Gradle cache 路径，识别旧版本依赖与无效缓存，并提供安全清理能力。

---

## 2. 需求是否可行

结论：**完全可行，而且具备明确使用价值。**

其可行性主要来自以下几点：

### 2.1 Maven 与 Gradle 本地缓存结构都具备可扫描性

Maven repository 的目录结构通常为：

```text
~/.m2/repository/groupId/artifactId/version/
```

例如：

```text
~/.m2/repository/org/springframework/spring-test/6.1.16/
```

Gradle 本地缓存虽然结构比 Maven 更复杂，但也有稳定的缓存目录，例如：

```text
~/.gradle/caches/
~/.gradle/wrapper/
```

在 Windows 中通常对应：

```text
C:\\Users\\<用户名>\\.gradle\\caches
C:\\Users\\<用户名>\\.gradle\\wrapper
```

因此程序依然可以稳定识别出：

- `groupId`
- `artifactId`
- `version`
- 缓存类型
- wrapper 分布

并进一步统计：

- 同一依赖存在多少个版本
- 每个版本占用多少空间
- 哪些版本较旧
- 哪些版本长期未使用
- 哪些缓存已经失效或可重建

### 2.2 旧版本清理本身具备明确收益

可直接带来：

- 释放磁盘空间
- 降低仓库臃肿程度
- 提升本地依赖管理可视化程度
- 帮助开发者理解哪些依赖长期堆积未再使用

### 2.3 该需求具有跨平台共性

无论是 macOS 还是 Windows，Maven 与 Gradle 的本地缓存机制都较稳定，区别主要在于默认路径、权限机制和交互方式，因此完全适合做成跨平台产品。

---

## 3. 产品核心定义

建议将该产品定义为：

> **一款扫描本机 Maven 本地仓库与 Gradle 本地缓存，识别旧版本、未使用依赖与可清理缓存，并提供安全清理建议的开发者工具。**

它不是单纯“删除文件”的工具，而应是一款：

- 懂 Maven 结构
- 懂 Gradle 缓存结构
- 懂版本关系
- 懂使用风险
- 支持预览与确认
- 支持跨平台

的本地依赖治理工具。

---

## 4. 为什么不能简单“只保留最新版本”

虽然这个需求可行，但不能采用“无脑删除旧版本”的策略。

### 4.1 不同项目可能依赖不同版本

例如：

- 项目 A 使用 `spring-test:6.1.14`
- 项目 B 使用 `spring-test:6.2.12`
- 项目 C 仍依赖 `5.3.29`
- 项目 D 的 Gradle 构建仍固定在某个 plugin / wrapper 版本

如果只保留最新版本，可能导致：

- 老项目重新构建时需要重新下载
- 公司内网私服不可用时构建失败
- 离线场景下构建失败

### 4.2 有些版本虽然旧，但仍在使用

版本“旧”不代表“无效”。  
因此产品必须提供的是：

- **旧版本识别**
- **未使用版本识别**
- **低风险清理建议**

而不是直接替用户做破坏性删除。

---

## 5. 产品功能设想

### 5.1 基础扫描能力

扫描本机 Maven repository 与 Gradle cache 路径，并识别：

- 仓库 / 缓存总大小
- 各 group / artifact 占用大小
- 每个 artifact 的版本列表
- 各版本最后修改时间
- 依赖重复版本数量
- Gradle modules / files / wrapper 分布
- 可清理缓存目录分类

### 5.2 清理建议能力

可基于规则给出清理建议，例如：

- 保留每个 artifact 最新 2 个版本
- 删除 180 天未访问版本
- 删除未被本机项目引用的版本
- 删除失败下载残留文件
- 删除 SNAPSHOT 历史残留
- 删除 Gradle 无效缓存
- 删除旧版 Gradle distribution / wrapper 缓存

### 5.3 删除执行能力

支持：

- Dry Run（预演，不真正删除）
- 手动勾选删除
- 一键执行清理
- 导出清理报告

### 5.4 安全保护能力

必须提供：

- 风险提示
- 删除前确认
- 可选备份 / 回收站策略
- 白名单机制
- 保留被本机项目引用的版本

---

## 6. 推荐的清理策略

建议产品支持多种规则组合，而不是单一规则。

### 6.1 按版本保留数量

例如：

- 每个 artifact 保留最新 2 个版本
- 删除更老版本

适合快速腾空间。

### 6.2 按本机项目引用情况保留

扫描本机项目中的：

- `pom.xml`
- `build.gradle`
- `build.gradle.kts`
- `settings.gradle`
- `settings.gradle.kts`
- `gradle.properties`

识别仍被项目使用的版本，并自动保护这些版本。

### 6.3 按最后访问时间清理

例如：

- 180 天未使用
- 365 天未使用

适合长期沉积依赖治理。

### 6.4 清理异常缓存

例如：

- `.lastUpdated`
- 下载失败残留
- 临时 metadata
- 过期 SNAPSHOT
- Gradle 临时缓存
- 旧 wrapper 压缩包
- 未使用的 transformed files 缓存

这类内容通常清理风险更低。

## 7. 技术风险与难点

### 7.1 Maven / Gradle 版本比较不能用简单字符串排序

例如：

- `5.2.9.RELEASE`
- `6.1.16`
- `7.0.2`
- `1.0.0-SNAPSHOT`

需要尽量遵循 Maven / Gradle 常见版本语义，而不是按字符串大小比较。

### 7.2 依赖目录不仅包含 jar

一个版本目录下可能包括：

- `.jar`
- `.pom`
- `sources.jar`
- `javadoc.jar`
- 校验文件
- metadata

清理时应按“版本目录”整体处理。

### 7.3 自定义路径与多缓存位置

用户本地仓库不一定都在默认位置，可能在 `settings.xml` 中配置了：

- 自定义 `localRepository`
- 公司统一磁盘路径
- 外接盘路径

Gradle 也可能存在：

- 自定义 `GRADLE_USER_HOME`
- CI / 本地环境不同缓存目录
- wrapper 与 caches 分离

所以产品应支持自动发现与手动指定路径。

### 7.4 离线与私服环境风险

如果企业私服不可访问，删除后依赖无法重新下载，可能影响构建。

因此需要：

- 风险提示
- 预演模式
- 删除清单导出
- 白名单保护

## 8. macOS 与 Windows 的实现考虑

该需求在两种机型上都可实现，但实现细节应分别考虑。

### 8.1 macOS 实现考虑

#### 默认路径

默认 Maven 仓库常见路径：

```text
~/.m2/repository
```

默认 Gradle 缓存常见路径：

```text
~/.gradle/caches
~/.gradle/wrapper
```

#### 适配点

- 读取 `~/.m2/settings.xml`
- 读取 `~/.gradle` 与 `GRADLE_USER_HOME`
- 适配用户目录和软链接
- 处理 Finder 中的空间展示与权限提示
- 支持扫描外接磁盘或自定义仓库 / 缓存目录

#### 适合的产品形态

- macOS 桌面应用
- CLI 工具
- JetBrains 插件

#### 删除实现建议

- 优先使用应用内删除逻辑
- 提供“移入回收站”而不是直接永久删除
- 对系统权限受限路径给出提示

### 8.2 Windows 实现考虑

#### 默认路径

默认 Maven 仓库常见路径：

```text
C:\\Users\\<用户名>\\.m2\\repository
```

默认 Gradle 缓存常见路径：

```text
C:\\Users\\<用户名>\\.gradle\\caches
C:\\Users\\<用户名>\\.gradle\\wrapper
```

#### 适配点

- 处理 Windows 路径分隔符
- 识别 `%USERPROFILE%\\.m2\\settings.xml`
- 识别 `%USERPROFILE%\\.gradle` 与 `GRADLE_USER_HOME`
- 兼容中文用户名、空格路径、长路径
- 处理文件占用、杀毒软件扫描、只读属性

#### 适合的产品形态

- Windows 桌面应用
- PowerShell / CLI 工具
- IntelliJ IDEA 插件

#### 删除实现建议

- 尽量支持回收站删除
- 删除失败时识别“文件占用中”
- 对被 IDE / Maven 进程占用的文件给出提示

### 8.3 跨平台统一能力

macOS 与 Windows 可以共用大部分核心逻辑：

- 仓库扫描
- Gradle 缓存扫描
- Maven 坐标识别
- Gradle 模块缓存识别
- 版本聚合
- 空间统计
- 清理规则引擎
- 项目引用分析

平台差异主要集中在：

- 默认路径发现
- 权限处理
- 删除方式
- UI 样式与交互

因此非常适合采用：

> **一套核心扫描引擎 + 两端平台适配层**

的方案。

## 9. 推荐产品形态

### 9.1 CLI 工具

适合工程师用户，开发成本低，MVP 落地快。

例如：

```bash
dep-clean scan
dep-clean suggest
dep-clean prune --keep 2 --days 180
```

优点：

- 易开发
- 易集成
- 跨平台方便

### 9.2 桌面应用

适合更广泛开发者用户，能直观看到：

- 哪些依赖最大
- 哪些版本重复最多
- 预计可释放多少空间

优点：

- 可视化强
- 上手门槛低
- 更像成熟产品

### 9.3 IDE 插件

适合作为增强形态，但不建议首版优先做。

原因：

- 宿主依赖更强
- 生命周期受 IDE 限制
- 实现复杂度高于独立工具

---

## 10. MVP 建议

建议第一版只做最有价值、风险最可控的能力。

### 10.1 MVP 功能清单

1. 自动发现 Maven repository 与 Gradle cache 路径
2. 支持手动指定扫描目录
3. 扫描 Maven artifact / version 与 Gradle cache 结构
4. 统计大小与重复版本
5. 标出“可清理候选项”
6. 支持保留最新 N 个版本
7. 支持 Dry Run
8. 手动确认后删除
9. 导出清理报告

### 10.2 首版暂不做

建议首版不优先做：

- 回滚恢复中心
- 远程私服联动分析
- 多语言 UI
- 企业级团队治理面板

---

## 11. 推荐技术方案

### 11.1 核心能力层

核心模块建议包括：

- Repository 路径发现器
- Maven 坐标解析器
- Gradle 缓存解析器
- 版本比较器
- 文件大小统计器
- 项目引用扫描器
- 清理策略引擎
- 删除执行器
- 报告生成器

### 11.2 技术路线建议

如果优先考虑跨平台与开发效率，可考虑：

- **后端核心**：Java / Kotlin / Go / Rust
- **桌面 UI**：Tauri、Electron、Flutter Desktop、JetBrains Compose Desktop
- **CLI**：Java / Kotlin CLI 或 Go CLI

若希望与 Java 生态结合更紧密，推荐：

> **Kotlin/Java 实现核心扫描引擎 + CLI 首发**

后续再扩展桌面端。

---

## 12. 结论

这个需求不仅可行，而且很有实际价值。

它解决的是 Java 开发者长期存在但缺乏好工具处理的问题：

- 本地 Maven 仓库臃肿
- 本地 Gradle 缓存臃肿
- 旧版本依赖堆积
- 空间占用不可见
- 清理缺乏安全策略

正确的产品方向不应是“暴力删除旧版本”，而应是：

> **做一款安全、可预览、可跨平台、理解 Maven 与 Gradle 规则的本地依赖清理工具。**

从落地路径上看，最推荐：

1. 先做 **CLI MVP**
2. 再做 **桌面可视化版本**
3. 后续再考虑 **IDE 插件化**

---

> 一款帮助开发者扫描并安全清理本机 Maven 本地仓库与 Gradle 本地缓存中旧版本依赖、无效缓存的跨平台工具，适用于 macOS 与 Windows。
